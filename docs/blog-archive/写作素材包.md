## 素材 1：生产环境分包异常自愈

```js
if (import.meta.env.PROD) {
  window.addEventListener('vite:preloadError', (event) => {
    event.preventDefault()
    window.location.reload()
  })
}
```

可讲点：

- 为什么线上会出现“旧 HTML + 新 chunk”错配
- 前端如何做“自愈式刷新”

常见误区：

- 只在控制台报错，不做用户侧恢复

一句话解释：

- “这段代码让页面在分包缓存错位时自动恢复，避免用户看到白屏后无从处理。”

```js
export default defineConfig(({ mode }) => ({
  plugins: [vue()],
  base: mode === 'offline' ? './' : '/quant-dashboard/',

```

一句话解释：

- “同一套代码，通过 `mode` 切换 `base`，实现了‘线上可部署、离线可打开’。”

## 素材 3：状态归一函数（防脏数据）

```js
function toNonNegativeInt(value) {
  const num = Number(value)
  if (!Number.isFinite(num) || num < 0) return 0
  return Math.floor(num)
}
```

可讲点：

- 输入永远不可信，先归一再计算
- 为什么统一入口比各处临时判断更稳

常见误区：

- 在模板里零散处理，导致口径不一致

一句话解释：

- “我把‘防御性编程’前置到状态层，避免异常输入污染后续所有计算。”

## 素材 5：规则引擎式计分（封顶与叠加）

```js
const codingDurationScore = Math.min(state.coding_work_units * 5, 30)
const codingExerciseScore = Math.min(state.coding_exercise_groups * 5, 30)
const codingKnowledgeScore = Math.min(state.coding_knowledge_units * 10, 50)
const codingBlogScore = state.coding_extra_blog * 10
const codingIssueOptScore = Math.min(state.coding_issue_opt_count * 5, 10)
const coding = Math.min(
  codingDurationScore + codingExerciseScore + codingKnowledgeScore + codingBlogScore + codingIssueOptScore,
  70
)
```

可讲点：

- 分项上限 + 总上限的双层约束
- 为什么“博客不封顶”但总分仍可控

常见误区：

- 只做分项封顶，不做总上限，导致系统可被刷分

一句话解释：

- “分项上限控制异常输入，总上限保证整体公平，这就是规则可维护的关键。”

function startAdjust(field, delta) {

## 素材 8：文本解析到结构化记录

```js
function parseRecordMap(text) {
  const lines = text.split('\n')
  const list = []
  let current = null

  lines.forEach((rawLine) => {
    const line = rawLine.trim()
    if (!line) return
    if (line.startsWith('[') && line.endsWith(']')) {
      if (current?.date) list.push(current)
      current = { date: line.slice(1, -1) }
      return
    }
    if (!current) return
    const [key, value] = line.split('=')
    if (!key || value === undefined) return
    current[key.trim()] = value.trim()
  })

  if (current?.date) list.push(current)
  return list
}
```

可讲点：

- 轻量 DSL（日期段 + key=value）解析
- 解析函数如何保持“容错但不纵容脏数据”

常见误区：

- 把解析和计算混在一起，后续难维护

一句话解释：

- “先解析成统一记录结构，再进入计算层，是降低复杂度的关键一步。”

## 素材 9：图表渲染与销毁

```js
if (chartInstance) {
  chartInstance.destroy()
  chartInstance = null
}

chartInstance = new Chart(chartCanvas.value, {
  type: 'line',
  data: { labels, datasets },
})
```

可讲点：

- 为什么重绘前必须 `destroy`
- 防内存泄漏与重复图层叠加

常见误区：

- 每次更新都 `new Chart`，但不销毁旧实例

一句话解释：

- “图表是有状态对象，重建前先销毁，才能保证渲染正确且不泄漏。”

## 素材 10：侦听 + `nextTick` 触发重渲染

```js
watch([records, selectedSeries], async () => {
  await nextTick()
  renderChart()
}, { deep: true, immediate: true })
```

可讲点：

- 数据变化与 DOM 更新时序
- `deep` 与 `immediate` 的实际用途

常见误区：

- 不等 DOM 就渲染，导致 canvas 为空或尺寸异常

一句话解释：

- “先等视图稳定，再渲染图表，能显著减少时序类 bug。”

# 从 0 实现计分规则引擎：Vue3 项目中的状态设计与上限控制（2）

## 摘要

这篇聚焦项目最核心的部分：`useProgressStore`。  
目标是回答：如何把“业务规则”写成可维护代码，并在规则迭代时不把页面搞乱。

---

## 1. 为什么先做规则层

在这个项目里，页面只是输入和展示，真正决定正确性的，是计分规则。  
如果规则散落在多个组件里，后续改分值会非常痛苦。

所以我先做了一个统一入口：

- 默认状态
- 输入归一
- 兼容旧字段
- 分数计算

---

## 2. 状态设计：先定义字段语义

我把每个字段都定义成“可解释的业务单位”，例如：

- `coding_work_units`：编程单元（30 分钟）
- `coding_exercise_groups`：练习题组
- `coding_knowledge_units`：知识学习单元
- `coding_issue_opt_count`：技术难点/优化细节合并计数

这样做的好处：

- UI 文案能对应字段
- 导出文本可读
- 统计页能复用同一语义

---

## 3. 输入归一：先防御，再计算

核心函数：

```js
function toNonNegativeInt(value) {
  const num = Number(value)
  if (!Number.isFinite(num) || num < 0) return 0
  return Math.floor(num)
}
```

为什么要统一归一：

- 用户输入可能是空字符串、小数、负数
- 导入文本也可能有脏数据
- 不归一会导致分数抖动或异常

---

## 4. 兼容旧字段：避免规则升级破坏历史数据

我把旧字段兼容写在 `normalizeIntoState` 里统一处理。  
典型例子是把旧版两个字段合并到新版一个字段：

```js
state.coding_issue_opt_count =
  toNonNegativeInt(source.coding_extra_debug) +
  toNonNegativeInt(source.coding_extra_opt)
```

原则：

- 新字段优先
- 旧字段兜底
- 映射逻辑单点维护

---

## 5. 计分公式：分项上限 + 总项上限

编程分数示意：

```js
const codingDurationScore = Math.min(state.coding_work_units * 5, 30)
const codingExerciseScore = Math.min(state.coding_exercise_groups * 5, 30)
const codingKnowledgeScore = Math.min(state.coding_knowledge_units * 10, 50)
const codingBlogScore = state.coding_extra_blog * 10
const codingIssueOptScore = Math.min(state.coding_issue_opt_count * 5, 10)
const coding = Math.min(
  codingDurationScore + codingExerciseScore + codingKnowledgeScore + codingBlogScore + codingIssueOptScore,
  70
)
```

这里有两个层级：

- 分项封顶：防极端输入
- 总分封顶：保证规则公平性

---

## 6. `computed` 作为规则引擎载体

我用 `computed` 管理 `scores`，好处是：

- 依赖字段变化时自动重算
- 页面不需要手动触发计算
- 逻辑集中，可测试性高

这类“纯函数 + 派生状态”的模式，非常适合工具类项目。

---

## 7. 给同类项目的建议

如果你也在做规则密集型页面：

1. 先定字段语义，再做 UI
2. 先定口径，再做交互
3. 输入归一一定写成统一函数
4. 兼容逻辑集中在一处

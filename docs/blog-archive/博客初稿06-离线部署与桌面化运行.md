# 不上 Electron，也能有桌面体验：Vite 离线部署与窗口化运行（6）

## 摘要

系列最后一篇，讲“可用性闭环”的最后一步：  
如何让一个前端项目在断网环境稳定运行，并尽量接近桌面应用体验。

---

## 1. 为什么要做离线能力

对这种个人工具来说，离线不是锦上添花，而是核心体验：

- 断网也能记录
- 无需依赖外部服务
- 跨设备复制文件即可用

---

## 2. 离线白屏的根因

Vite 项目常见问题是：资源路径跟部署环境不匹配。

- GitHub Pages：通常是子路径（如 `/quant-dashboard/`）
- 本地离线：需要相对路径（`./`）

如果把 `base` 写死，离线打开就会找不到 JS/CSS。

---

## 3. 我的解决方案：按 `mode` 切换 `base`

```js
export default defineConfig(({ mode }) => ({
  base: mode === 'offline' ? './' : '/quant-dashboard/',
}))
```

然后加一个脚本：

- `npm run build:offline`

这样同一套代码可兼容线上与离线构建。

---

## 4. 本地运行脚本：`start-offline.bat`

这个脚本做了四步：

1. 检查 npm
2. 构建离线包
3. 启动本地预览服务
4. 打开本地地址

优点是：用户不需要记命令，一次双击即可运行。

---

## 5. 静默独立窗口：`start-offline-silent.vbs`

我进一步做了“类桌面应用”体验：

- 静默启动服务（不弹命令行）
- 使用浏览器 `--app` 模式打开
- 默认 `600x700` 居中

这不是原生桌面应用，但对个人工具已经足够好用，且开发成本很低。

---

## 6. 这个方案的边界

它解决了“可离线 + 独立窗口体验”，但不是完整替代 Electron：

- 优点：轻量、快、维护成本低
- 局限：依赖系统浏览器、能力受浏览器安全策略限制

如果后续需要系统级能力（文件系统深度集成、托盘等），再考虑 Electron/Tauri。

---

## 7. 系列总结

这个项目从需求到落地，核心不是“技术炫酷”，而是三件事：

1. 规则统一
2. 数据可迁移
3. 使用成本低

前端小工具只要把这三件事做好，价值会远高于堆技术栈。

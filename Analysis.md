# quant-dashboard 项目分析

## 1. 项目概览
- **项目类型**：前端单页应用（SPA），用于记录「每日量化进度」并自动计算分数。
- **技术栈**：`Vue 3` + `Vite`，无路由、无状态管理库（使用自定义 store）。
- **当前成熟度**：已具备可运行的 MVP（最小可用产品）形态，核心流程（录入、计分、本地保存、导入导出）已实现。

## 2. 目录与模块结构
- `src/main.js`：应用入口，挂载 `App.vue`。
- `src/App.vue`：主页面与交互逻辑，包含全部表单、操作按钮、导入导出逻辑。
- `src/stores/useProgressStore.js`：业务状态与分数计算、`localStorage` 持久化逻辑。
- `src/style.css`：全局样式（当前保留较多 Vite 模板默认样式）。
- `src/components/HelloWorld.vue`：模板遗留组件，当前未在业务中使用。

## 3. 业务能力现状

### 3.1 已实现功能
- **日期维度记录**：通过 `date` 选择当天记录，并按 `progress_日期` 进行本地存储。
- **必做项计分**：
  - 英语固定分项（复选框）。
  - 编程时长与加分项（数值输入，含上限控制）。
- **可选项计分**：超额编程、绘画、文学创作，支持总分封顶。
- **自动保存**：监听 `state` 深度变化后自动写入 `localStorage`。
- **数据导出/导入**：支持文本格式导出到剪贴板，并可粘贴回应用导入。
- **睡前结算勾选**：支持目标复选记录（不参与得分计算）。

### 3.2 计分规则实现（当前行为）
- 英语：固定四项，总计最高 `60` 分。
- 编程：深度时长按每 30 分钟 5 分计，时长项上限 `30`；加上博客/难点/优化后整体封顶 `40`。
- 可选项：代码内显示时使用 `optionalTotal`（未封顶值）；总分时按 `Math.min(optionalTotal, 60)` 封顶。
- 总分：`english + coding + min(optional, 60)`，理论上最高 `160`。

## 4. 代码质量与风险评估

### 4.1 优点
- **逻辑集中**：业务核心在 `useProgressStore` 中统一管理，便于理解和后续抽离。
- **实现直接**：计分与存储逻辑清晰，便于快速迭代。
- **低依赖**：依赖少、启动快，适合作为轻量工具应用。

### 4.2 当前问题与潜在风险
- **导入字段类型不一致风险**：英语项在状态里以数字 `0/20` 存储，但导入逻辑按 `Number(value)` 处理，当前虽可用，但与复选框 `true-value/false-value` 绑定方式耦合较紧。
- **日期切换加载时机较脆弱**：仅在 `@change` 触发加载；首次进入页面不会主动读取当天已有数据（除非后续手动触发）。
- **可选部分展示与总分封顶语义不一致**：界面展示 `可选部分: optional / 60`，但 `optional` 目前是未封顶值，可能出现“显示超过 60”的认知偏差。
- **样式冲突隐患**：全局 `style.css` 仍保留 Vite 模板的深色主题、居中布局和按钮样式，与业务页面样式可能互相影响。
- **无输入约束细化**：数值输入虽设置 `min=0`，但未统一处理异常值（如超大值、小数、手动输入非预期格式）对统计的影响。
- **遗留文件未清理**：`HelloWorld.vue` 未使用，增加维护噪音。

## 5. 可维护性评估
- **模块划分**：当前单页面可接受，但 `App.vue` 已承载过多 UI 与业务逻辑，后续复杂度提升时可读性会下降。
- **可扩展性**：如需增加统计图、周/月报、云同步，建议尽早拆分 composable 与组件层。
- **测试覆盖**：暂无自动化测试，计分规则回归依赖手工验证，存在改动后回归风险。

## 6. 优化建议（按优先级）

### P0（建议优先）
1. **修正可选项展示口径**：展示值与总分口径统一（展示封顶后值，或明确同时展示“原始值/封顶值”）。
2. **初始化加载当天数据**：应用启动时执行一次 `loadFromStorage(state.date)`，避免已有数据不自动回填。
3. **输入值标准化**：对所有数值输入做统一归一（非负整数、异常值兜底）。

### P1（近期可做）
1. **拆分页面组件**：按模块拆为英语区、编程区、可选区、汇总区、导入导出区。
2. **抽离导入导出协议层**：定义清晰的字段映射与版本号，降低后续改字段的兼容成本。
3. **清理模板遗留内容**：移除未使用组件和默认样式，减少噪声。

### P2（中期规划）
1. **增加测试**：至少覆盖计分函数与导入解析函数的单元测试。
2. **增强数据能力**：增加周/月统计和历史趋势展示。
3. **可选云同步**：支持后端或第三方存储，避免本地数据丢失。

## 7. 结论
该项目已具备明确、可运行的核心价值：帮助用户以结构化方式记录并量化每日学习/工作进展。当前主要短板集中在展示口径一致性、初始化加载、输入健壮性和工程化基础（拆分与测试）。若按上述 P0/P1 逐步优化，可较快从“可用工具”提升为“稳定可维护的小型产品”。
